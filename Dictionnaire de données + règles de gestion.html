REGLES DE GESTION
30
RG1
Chaque enregistrement dans la table adresse doit avoir un identifiant unique (id). Cet identifiant ne peut pas être modifié une fois créé.
31
RG2
Lorsqu’un code FANTOIR est disponible, il doit correspondre au code INSEE de la commune de rattachement. Si aucun code FANTOIR n’est connu, un identifiant temporaire de 6 caractères alphanumériques est généré.
32
RG3
Le champ code_insee doit toujours correspondre à une commune existante dans la table commune. S’il existe un code_insee_ancienne_commune, alors nom_ancienne_commune doit aussi être renseigné.
33
RG4
code_postal et code_insee sont composés de 5 chiffres.
34
RG5
Le champ numero doit être un entier positif. Il peut être nul si l’adresse ne comporte pas de numéro (ex : lieu-dit).
35
RG6
Le champ nom_afnor doit être en majuscules selon la norme postale AFNOR XP Z 10-011.
36
RG7
Chaque adresse doit posséder une position unique (id_position).
37
RG8
Les coordonnées lon et lat doivent respecter les bornes WGS84 : longitude entre -180 et 180, latitude entre -90 et 90.
38
RG9
Le champ source_position doit prendre une valeur parmi : ('commune', 'cadastre', 'ign', 'arcep', 'inconnue').
39
RG10
Le champ type_position doit être cohérent avec la spécification BAL 1.3 (ex : entrée, segment, bâtiment).
40
RG11
Le champ certification_commune vaut 1 si la commune a validé l’adresse, sinon 0.
41
RG12
Le champ cad_parcelles doit respecter le format cadastral (14 caractères). Mais il peut y avoir plusieurs numéro séparés d'un "|". (Vu avec erreur à l'importation du .csv : trop court avec varchar(255)).
42
